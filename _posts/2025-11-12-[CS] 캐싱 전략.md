# # 캐싱 전략

캐싱 전략은 시스템의 성능을 향상시키기 위해 자주 사용되는 데이터를 임시 저장소에 보관하는 방법을 의미한다. 이를 통해 데이터 접근 시간을 단축시키고, 서버 부하를 줄일 수 있습니다. 캐싱 전략에는 여러 가지가 있으며, 주요 전략은 다음과 같다.

<br />

### # Cache Aside(Lazy Loading) 방식

Cache Aside 방식은 애플리케이션이 데이터베이스에서 데이터를 요청할 때 먼저 캐시를 확인하고, 캐시에 데이터가 없으면 데이터베이스에서 데이터를 가져와 캐시에 저장하는 방식이다. 이 방식은 읽기 작업이 많은 애플리케이션에 적합하다.


- **장점** :
    - 실제 요청된 데이터만 캐시에 저장되므로 불필요한 데이터 캐싱을 줄일 수 있다. 또한, 
    - 캐시에 문제가 발생해도 애플리케이션은 원본 데이터베이스에 직접 접근할 수 있기 때문에 서비스가 계속 작동할 수 있다.


- **단점** :
    - 캐시에 데이터가 없는 경우 데이터베이스에 직접 접근해야 하므로 초기 요청 시 지연이 발생할 수 있다. 
    - 캐시와 데이터베이스 간의 데이터 일관성을 유지하는 데 추가적인 관리가 필요하다.

<br />

### # 캐시 불일치를 해소할 수 있는 쓰기 전략

#### 1. Write Through 방식

Write Through 방식은 데이터가 데이터베이스에 쓰여질 때 동시에 캐시에도 데이터를 저장하는 방식이다. 이 방식은 데이터 일관성을 유지하는 데 유리하지만, 쓰기 작업이 느려질 수 있다.

#### 2. Write Back 방식

Write Back 방식은 데이터가 캐시에 먼저 쓰여지고, 일정 시간이 지난 후에 데이터베이스에 반영되는 방식이다. 이 방식은 쓰기 작업의 성능을 향상시키지만, 데이터 일관성 문제가 발생할 수 있다.

#### 3. Cache Invalidation 방식

Cache Invalidation 방식은 데이터가 변경될 때 해당 데이터에 대한 캐시를 무효화하는 방식이다. 이 방식은 데이터 일관성을 유지하는 데 효과적이지만, 캐시 미스가 발생할 가능성이 높아진다.
