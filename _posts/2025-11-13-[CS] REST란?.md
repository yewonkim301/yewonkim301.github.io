# # REST란?

REST(Representational State Transfer)는 웹 아키텍처 스타일 중 하나로, 클라이언트와 서버 간의 통신을 효율적으로 관리하기 위한 설계 원칙을 제공한다. REST는 자원을 중심으로 설계되며, HTTP 프로토콜을 기반으로 한다.

<br />

### # REST의 특징

1. **자원(Resource)**: REST에서는 모든 것이 자원으로 간주된다. 각 자원은 고유한 URI(Uniform Resource Identifier)를 통해 식별된다.

2. **표현(Representation)**: 자원은 다양한 형식(JSON, XML 등)으로 표현될 수 있다. 클라이언트는 서버로부터 자원의 표현을 받아서 처리한다.

3. **무상태성(Statelessness)**: RESTful 시스템에서는 각 요청이 독립적이며, 서버는 클라이언트의 상태를 저장하지 않는다. 모든 필요한 정보는 요청에 포함되어야 한다.

4. **캐시 가능성(Cacheability)**: 응답은 캐시될 수 있으며, 이를 통해 성능을 향상시킬 수 있다.

5. **계층화된 시스템(Layered System)**: 클라이언트는 중간 서버를 통해 자원에 접근할 수 있으며, 이는 보안과 확장성을 향상시킨다.

6. **일관된 인터페이스(Uniform Interface)**: REST는 일관된 인터페이스를 제공하여 클라이언트와 서버 간의 상호작용을 단순화한다. 이는 자원 식별, 자원 조작, 자기 설명 메시지, 하이퍼미디어를 통한 애플리케이션 상태 전이 등을 포함한다.

<br/>

### # REST의 장점

- **확장성(Scalability)**: RESTful 시스템은 클라이언트와 서버 간의 독립성을 유지하여 확장성을 높인다.

- **성능(Performance)**: 캐싱과 무상태성으로 인해 성능이 향상된다.

- **유지보수성(Maintainability)**: 일관된 인터페이스와 명확한 자원 식별로 인해 유지보수가 용이하다.

<br />

### # REST의 단점

- **복잡성(Complexity)**: RESTful 시스템을 설계하고 구현하는 데에는 일정 수준의 복잡성이 존재한다.

- **제한된 기능(Limited Functionality)**: REST는 특정 기능(예: 트랜잭션 관리 등)을 제공하지 않으며, 이를 구현하기 위해 추가적인 설계가 필요할 수 있다.

<br />

### # RESTful API

RESTful API는 REST 원칙을 따르는 웹 API를 의미한다. RESTful API는 HTTP 메서드(GET, POST, PUT, DELETE 등)를 사용하여 자원을 조작하며, URI를 통해 자원을 식별한다. RESTful API는 확장성, 성능, 유지보수성 측면에서 많은 장점을 제공한다.
